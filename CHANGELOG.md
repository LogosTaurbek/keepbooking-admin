## [Unreleased] — 2026-07-14

### Added — Фото ресторана
- `RestaurantPhotosPage` (`/restaurants/:id/photos`) — сетка миниатюр, кнопка "Upload photo" (скрытый `<input type="file">`, триггерится программным кликом), удаление по кнопке, всплывающей при наведении на карточку
- Бэкенд для этого оказался уже полностью готов и подключён к реальному MinIO (S3-совместимое хранилище) — настоящий multipart-upload через `FileStorageService`/AWS SDK v2 `S3Client`, публичные URL, валидация типа/размера файла на сервере. Бэкенд-работы не потребовалось вообще, только фронтенд
- `api/client.ts`: `apiFetch` расширен поддержкой `FormData` в теле запроса — раньше всегда делал `JSON.stringify` и жёстко ставил `Content-Type: application/json`; теперь при `body instanceof FormData` заголовок `Content-Type` не выставляется вручную (браузер сам проставляет `multipart/form-data` с нужным boundary) и `JSON.stringify` не вызывается. Все существующие JSON-запросы поведение не поменяли
- Осознанно не реализовано на этом срезе (у бэкенда нет соответствующих эндпоинтов — тот же паттерн, что и с drag&drop для столов ранее): выбор обложки/primary-фото, изменение порядка фотографий. `position` на бэкенде выставляется только при загрузке (append-to-end), эндпоинта для reorder нет
- Проверено вживую через headless Chrome (Playwright, временно установлен и удалён) на реальном бэкенде и реальном MinIO: логин → переход на страницу фото → загрузка настоящего JPEG-файла → подтверждение, что `<img>` действительно догрузился (`naturalWidth > 0`, не битая ссылка) → удаление фото → пустое состояние вернулось

### Changed — Рабочие часы: сохранение по одному дню вместо всей недели
- Бэкенд перешёл с full-replace `PUT /working-hours` на per-day `PATCH /working-hours/{dayOfWeek}` (подробности и причина — в `CHANGELOG.md` бэкенда за эту же дату). `RestaurantWorkingHoursPage` переделана под это: единой кнопки "Save week" больше нет, у каждой строки дня — своя кнопка "Save", свой индикатор "Saving…" и своя строка ошибки под ней (не общий баннер) — редактирование и сохранение одного дня больше не трогает остальные шесть
- `api/workingHours.ts`: `replaceWorkingHours(restaurantId, items[])` заменён на `upsertWorkingHoursDay(restaurantId, dayOfWeek, request)`; тип `WorkingHoursItemRequest` заменён на `UpsertWorkingHoursDayRequest` (без `dayOfWeek` в теле, зеркалит новый бэкенд-DTO)
- Проверено вживую через headless Chrome (Playwright, временно установлен и удалён) на реальном бэкенде: изменили и сохранили только вторник (Tuesday) → перезагрузка страницы → значение вторника персистнуло, остальные дни не пострадали

- `RestaurantAnalyticsPage` (`/restaurants/:id/analytics`) — карточки с суммарными метриками за выбранный диапазон дат (брони по статусам, confirmation rate, уникальные гости) + топ-5 популярных часов и топ-5 столов в виде горизонтальных барчартов на чистом Tailwind (без графической библиотеки)
- Диапазон дат по умолчанию — последние 30 дней, есть форма "From/To" + кнопка Apply для произвольного периода (`api/analytics.ts`, зеркалит `RestaurantAnalyticsDto`)

### Added — Рабочие часы ресторана
- `RestaurantWorkingHoursPage` (`/restaurants/:id/hours`) — две секции: недельное расписание (7 строк Пн–Вс) и даты-исключения (праздники/особые часы), зеркалит бэкендовые `restaurant_working_hours` и `restaurant_working_hours_overrides` через `api/workingHours.ts`
- Недельное расписание сохраняется одним `PUT /working-hours` с полным списком из 7 дней — бэкенд делает full-replace (`replaceWeek` сначала удаляет все строки ресторана, потом вставляет присланные), поэтому форма всегда отправляет все 7 дней разом, а не PATCH одного дня; `dayOfWeek` на бэкенде — просто `Integer` 1–7 (1=Monday) без энам, маппинг в человекочитаемые названия дней сделан на фронте
- Даты-исключения: создание/редактирование через upsert-by-date (`PUT /working-hours/overrides`), удаление возвращает дату к недельному расписанию (`DELETE /working-hours/overrides/{date}`)
- Бэкенд-гэпов не найдено — `WorkingHoursController`/`WorkingHoursService` уже имели полный CRUD с той же проверкой владения, что и в `HallService`/`MenuItemService`. Заодно подтвердилось: `BookingService.create()` действительно проверяет рабочие часы через `WorkingHoursResolver` и по умолчанию считает ресторан без настроенного расписания "always closed" — так что эта страница не косметическая, без неё вообще нельзя принимать брони
- Проверено вживую через headless Chrome (Playwright, временно установлен и удалён) на реальном бэкенде: логин → переход на страницу часов → снятие Closed с понедельника, установка 09:00–22:00, сохранение недели → перезагрузка страницы, значения персистнули → добавление override на 2026-12-25 как Closed → удаление override
